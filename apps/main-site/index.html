CI/CD Pipeline

Every push triggers a five-stage pipeline that detects what changed, validates code quality, builds production bundles, runs end-to-end tests, and deploys — automatically.

1

Change Detection

The pipeline diffs the commit to identify which apps and packages changed. Shared package changes cascade to all dependent apps. Only affected apps proceed through the remaining stages.

2

Validation Gates

Five checks run in parallel: Prettier format verification, ESLint, TypeScript type checking, Vitest unit tests with an 80% coverage threshold, and an npm security audit. All five must pass before the pipeline continues.

3

Build

Each affected SPA produces an optimized production bundle via Vite. Build artifacts are uploaded and reused in later stages so nothing is built twice.

4

E2E Tests

All three apps are served via Docker and nginx, mirroring production routing. Playwright runs 28 tests on Desktop Chrome and Mobile Pixel 5 — covering functionality, accessibility (axe-core WCAG), and visual regression.

Deploy

End-to-End Testing

E2E tests use a real browser to interact with the app the way a user would — clicking links, reading text, navigating between pages. Unlike unit tests that check isolated functions, E2E tests verify that everything works together: the server, the routing, the rendered UI, and the user flow. This project runs 28 Playwright tests across 5 testing categories.

Test Environment

Tests run against a Docker container with nginx serving all three apps on port 8080 — the same routing rules, SPA fallbacks, and security headers used in production. This matters because tests that skip the real server miss entire categories of bugs: a misconfigured nginx route, a missing SPA fallback, or a broken path prefix. By testing against the real infrastructure, E2E catches what unit tests and dev servers cannot.

Smoke Tests 3 tests

Navigate to each app's URL and check that the server responds with HTTP 200.

Why this matters: The most basic test possible — if the server returns an error code, nothing else matters. Smoke tests catch broken builds, missing files, and misconfigured nginx routing before any other test runs. They answer one question: "Is the app reachable?"

How it works: page.goto("/admin/") opens a real Chromium browser to that URL. response.status() reads the HTTP status code from the server's response. The test asserts it equals 200.

Content Verification 10 tests

After each page loads, assert that specific headings, text, and UI elements are visible in the browser.

Why this matters: A page can return HTTP 200 but render nothing — a blank white screen with an empty <div id="root">. Content verification proves that React actually mounted, the correct component rendered, and real data is displayed. If the heading reads "Admin Dashboard", it means the router matched the right route, the component imported successfully, and the DOM is populated.

How it works: page.locator("h1") finds elements by CSS selector. toHaveText("Admin Dashboard") asserts the element's text content. page.getByText("Total Users") finds elements by visible text content. toBeVisible() confirms the element is rendered and not hidden.

Navigation Testing 10 tests

Click links and buttons in the UI, then verify the correct page loads with the expected content.

Why this matters: Navigation tests verify that the router actually works — that clicking "Users" in the sidebar navigates to the Users page and renders real content. This catches broken <Link> components, wrong href values, misconfigured basename in React Router, and nginx routing issues between apps. The cross-app tests are especially important: they prove a user can click from the static main site into a React SPA and land on the correct page, which requires nginx, the SPA's router, and the build output to all agree on the URL structure.

How it works: page.getByRole("link", { name: "Users" }) finds a link by its accessible name (what a screen reader would announce), not a fragile CSS selector. .click() simulates a real mouse click. Then we assert the new page heading appeared, confirming navigation completed.

Accessibility Testing 2 tests

Run the axe-core accessibility engine against each SPA's fully rendered page and assert zero WCAG violations.

Why this matters: Automated accessibility testing catches common issues that make apps unusable for people with disabilities — missing alt text on images, insufficient color contrast between text and background, form inputs without labels, and incorrect ARIA attributes. These tests check against WCAG 2.0 A/AA and WCAG 2.1 A/AA standards. They won't catch everything (you still need manual testing for keyboard navigation and screen reader flow), but they catch the most common violations automatically on every commit, preventing regressions.

How it works: The @axe-core/playwright package injects the axe accessibility engine into the page after it renders. new AxeBuilder({ page }) creates a scanner. .withTags(["wcag2a", "wcag2aa", "wcag21a", "wcag21aa"]) configures which WCAG rules to check (about 80 rules total). .analyze() scans every element in the DOM. The test asserts results.violations is an empty array — any violation fails the build.
  • has no accessibility violations admin Scans the admin dashboard for WCAG violations after React renders all components
  • has no accessibility violations portal Scans the portal dashboard for WCAG violations after React renders all components

Visual Regression Testing 3 tests

Take a full-page screenshot and compare it pixel-by-pixel against a saved baseline image.

Why this matters: Visual regression catches bugs that no other test can detect — a CSS change that shifts a button off-screen, a font that stopped loading, a layout that breaks only on mobile. These are invisible to content-based tests because the text is still "correct" even when it's rendered in the wrong place. If more than 1% of pixels differ from the saved baseline, the test fails. Baselines are updated intentionally by running --update-snapshots when visual changes are expected.

How it works: page.toHaveScreenshot("name.png", { maxDiffPixelRatio: 0.01 }) captures a full-page screenshot and compares it to a stored baseline PNG. If the images differ by more than 1% of pixels, the test fails and Playwright saves an "actual", "expected", and "diff" image so you can see exactly what changed. Each test runs on both Desktop Chrome and Mobile viewports, so a total of 6 screenshot comparisons happen per run.

  • visual regression - admin dashboard Screenshots the admin SPA dashboard on desktop and mobile
  • visual regression - portal dashboard Screenshots the portal SPA dashboard on desktop and mobile

Multi-Device Coverage

Every test above runs twice: once on Desktop Chrome (1280×720) and once on Mobile Pixel 5 (393×851). That's 56 total test runs from 28 test definitions. This catches responsive layout bugs, touch-target sizing issues, and viewport-specific rendering problems. Configured in the Playwright projects array.

Run E2E tests locally: pnpm test:e2e

Update visual baselines after intentional changes: pnpm test:e2e -- --update-snapshots

Start all apps and run everything: pnpm run-all
